Troubleshooting remote connection issues

  • If you try to connect Wwise to your game, but the computer/console does not appear in the Remote Connections dialog, do the following:

    • Verify that the sound engine is in either debug or profile. You can't connect to a release build of the sound engine.

    • Verify that the Communications module is properly initialized in your game. For more information, refer to the "Initializing Communications" section of the Wwise SDK documentation.

    • Make sure the same port number is specified for the "Game Discovery Broadcast Port" in your project settings and in the AkCommSettings structure used in your game to initialize communications. For more information, refer to Specifying network ports and the "Initializing Communications" section of the Wwise SDK documentation.

    • Verify that the computer/console is properly connected to the network. Note that some consoles have multiple network adapters, including one for debugging and one for regular network traffic. Make sure to connect all of them before trying to re-connect from Wwise.

    • Verify that the computer/console is on the same subnet as the computer running Wwise.

    • Check if an active firewall is blocking the connection. If you have an active firewall on the computer running Wwise, try adding the Wwise application to the list of exceptions in your firewall (refer to your firewall's documentation for instructions). If you need to change the ports used for Wwise communication (including using fixed ports instead of dynamic ports), refer to Specifying network ports and the "Initializing Communications" section of the Wwise SDK documentation.

    • A port might be blocked by Windows. The Game Discovery Broadcast Port is a UDP connection. Check that this port and type of connection isn't blocked by your network or machine. This port might be somewhere other than on your computer.

    • If you know the IP address of the remote computer/console, you can click Connect to IP in the Remote Connections dialog and enter the IP address manually. If you can connect to the game in this manner, then it's only the broadcast from the remote computer/console that is blocked. After connecting a computer/console once, it will automatically appear in the History tab of the Remote Connections dialog, so you won't have to enter the IP address when you try to reconnect in the future.

    • Verify if Wwise or the network might be causing a connection failure. To test if this is the cause, set a known open port to the Command Port in the game Comm::init and run the game. Make sure the call to Comm::init initializes successfully. On the Windows machine you are connecting from, open a PowerShell window and type the following command:

      Test-NetConnection -computername <ipAddress> -port <commandPort>

      Where <ipAddress> is the IP address or the name of the machine the game is running on. The <commandPort> is the Command Port specified, for example, port 24025.

      The result should show TcpTestSucceeded : True. If that isn't the result, something is blocking that port for that TCP connection.

  • If you try to connect to your game from a very large project and Wwise seems to hang on the connect screen, try the following:

    • Open your project, try emptying the current Soundcaster and Mixing Desk sessions, remove what is loaded into the Transport Control, and then try to connect to the game with Profile and Edit (Sync Inspected Objects Only) instead of Profile and Edit (Sync All Modified Objects).

      Note

      When Wwise connects to a game with Profile and Edit (Sync All Modified Objects) selected, it pushes to the game all information needed to ensure what is displayed in Wwise is in sync with what is in the game. This can take some time and use more memory in the game. But, with Profile and Edit (Sync Inspected Objects Only), Wwise does not push everything. Instead, Wwise pushes only what is selected in the Property Editor, Transport Control, and what is present in the active Soundcaster and Mixing Desk sessions. If you are trying to connect to your game from a Wwise project that has an active Soundcaster or Mixing Desk session containing a large amount of data, you may experience this problem.

      Also, modified media (sounds, MIDI, Convolution IR, etc.) that is currently in use by the game will be transferred to the game too. This can also increase the time it takes to start profiling the game.

      If the remote connection is only for profiling purposes, you can also connect using Profile Only which will prevent any synchronization between the Authoring tool and the game. Note that you will not be able to play sounds in the Authoring tool when connected with this option.
