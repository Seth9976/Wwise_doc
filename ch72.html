ReadSpeaker speechEngine

The ReadSpeaker speechEngine plug-in provides text-to-speech support for your Wwise projects. This plug-in uses a proprietary text-to-speech engine optimized for performance and memory footprint, which ensures smooth integration and efficient resource usage. It processes text and generates audio on the device in real time to support dynamic dialogue, narration, reading of GUI elements, and so on.

The ReadSpeaker speechEngine plug-in provides the following features and benefits:

  • Enhances accessibility in your games by providing narration of on-screen UI elements.

  • Prototypes voice acting during early stages of development.

  • Supports speech-enabled in-game chat messages in multiplayer games and more.

  • Reads AI-generated text and dialog aloud with effectively zero latency.

  • Makes localization easy with the many included languages, each recorded by native speaking voice actors.

Voices are recorded and processed only by ReadSpeaker, using full-disclosure contracts with the voice actors.

Note

On Nintendo Switch and PS4, the voices sound slightly different than they do in Wwise Authoring and on other platforms because different voice models are used on those platforms.

Installation

Install the speechEngine plug-in through the Audiokinetic Launcher. See Installing Plug-ins for more information.

After installing the plug-in, add it to your project as a child in the Containers hierarchy. You can also add it as a sound source, or link the speechEngine objects to events, game syncs, and so on, as you would with any other object.

Runtime Dependencies

At runtime, the plug-in dynamically loads a language-specific library for each voice in use. These libraries follow the vt_[lang] naming convention (for example, vt_eng.dll for English on Windows or vt_bre.so for British English on Android). They must be on a path that the system's DLL load operation can locate, typically in the same directory as the speechEngine library.

Loading the Plug-in at Runtime

The plug-in loading method depends on whether your project links to it dynamically or statically.

  • Dynamic loading: Call AK::SoundEngine::RegisterPluginDLL with the name "SpeechEngine" before posting any speechEngine events. The corresponding DLL or shared library must be on a path that the operating system's dynamic linker can locate.

    AK::SoundEngine::RegisterPluginDLL(AKTEXT("SpeechEngine"));

  • Static linking: Include the plug-in factory header. This registers the plug-in automatically at startup without an explicit RegisterPluginDLL call.

    #include <AK/Plugin/SpeechEngineSourceFactory.h>

    When linking statically, you must also link against the libvtapi library distributed with the plug-in. For example, for the Windows 64-bit vc170 platform, the library is located at SDK\x64_vc170\Release\lib\libvtapi.lib.

Sending Text During Runtime

You can set text in Wwise Authoring in the plug-in's Text parameter. When text is also sent from game code via AK::SoundEngine::SendPluginCustomGameData, the game code text takes precedence over the value set in Wwise Authoring.

To send text from game code, call AK::SoundEngine::SendPluginCustomGameData using the following Company ID and Plug-in ID to target the speechEngine source:

in_uCompanyID = 312
in_uPluginID = 1

Call PostEvent first, then SendPluginCustomGameData immediately after, in the same game loop iteration. The plug-in reads the custom game data during its initialization, and it is dispatched asynchronously after the event is posted. Send the data before the next audio rendering frame to ensure the plug-in receives it in time.

The text must use UTF-8 character encoding. If no text is available — neither from Wwise Authoring nor from game code — the plug-in logs a message to the Wwise Monitor and produces no audio output.

Example: 

const AkUInt32 kCompanyID = 312;
const AkUInt32 kPluginID = 1;
const AkGameObjectID GAME_OBJECT_SPEAKER = 100;

std::string strTextToSynthesize = "This voice is generated using on-device text-to-speech technology.";

// 1. Post the event to start audio playback.
AkPlayingID playingID = AK::SoundEngine::PostEvent(
    "Play_speechEngine",
    GAME_OBJECT_SPEAKER,
    AK_EndOfEvent | AK_Marker,
    &MyMarkersCallback,
    pCookie);

// 2. Immediately send the text to synthesize.
AK::SoundEngine::SendPluginCustomGameData(
    AK_INVALID_UNIQUE_ID,
    GAME_OBJECT_SPEAKER,
    AkPluginTypeSource,
    kCompanyID,
    kPluginID,
    (void*)strTextToSynthesize.c_str(),
    static_cast<AkUInt32>(strTextToSynthesize.length()) + 1);

For a complete example in C++, refer to DemoSpeechEngine.cpp as part of the IntegrationDemo project.

Speech Synthesis Markup Language (SSML)

The plug-in supports SSML, which allows you to control pronunciation, emphasis, pauses, and other speech characteristics by embedding markup tags directly in the text. SSML tags are structured like XML and consist of a tag name and optional attributes enclosed in angled brackets (<>).

For example:

<emphasis level="strong">This is important.</emphasis>

To enable SSML processing, select the SSML toggle on the plug-in object in Wwise Authoring. When enabled, this setting applies to both the text from Wwise Authoring and to text sent by SendPluginCustomGameData at runtime. Enabling SSML has a minor negative effect on performance during speech synthesis, even if no tags are present in the text.

For a full reference of supported SSML tags, see Speech Synthesis Markup Language (SSML).

Note

  • The plug-in does not support <audio> or <voice> tags.

  • If the SSML text contains a syntax error, the synthesis fails. An error message is logged to the Wwise Monitor and no audio is produced. Validate your SSML markup before shipping.

Accessing Generated Word Information

When generating speech, the plug-in produces audio markers alongside the audio data. These markers carry timing and linguistic information, such as word boundaries and phoneme transitions, that you can use to drive in-game visuals, such as highlighted subtitles or lip-sync animation. For more information on audio markers in Wwise, see Integrating Markers.

To receive audio markers, pass AK_Marker as a callback flag when calling PostEvent, and handle AK_Marker callbacks in your callback function.

Each audio marker has the following fields:

  • Marker ID: identifies the type of text-to-speech event the marker represents (see the following table).

  • Position: the position in the audio stream where the event occurs, in samples.

  • Label: a string containing additional data whose content depends on the marker type.

Marker Types

Marker Type

Marker ID

Label Content

Begin Word

'bwrd'

Two integers separated by a space: the zero-based start byte index and the inclusive end byte index of the word in the synthesized text, based on its UTF-8 representation. Use them to extract or highlight the word being spoken.

Begin Phoneme

'bphm'

The viseme ID for the phoneme, encoded as a decimal integer string. Use this to drive mouth animation for the speaking character.

For more information on interpreting the viseme IDs, refer to this page.

End Phoneme

'ephm'

Empty. Signals the end of the current phoneme.

End Word

'ewrd'

Empty. Signals the end of the current word.


When the synthesized speech is played, audio markers are always received in the following pattern:

  1. BEGIN WORD

  2. BEGIN PHONEME

  3. END PHONEME

  4. ...

  5. BEGIN PHONEME

  6. END PHONEME

  7. END WORD

The number of Begin Phoneme / End Phoneme pairs within a word depends on the word's length. At the end of the final word in the sentence, the synthesizer emits one additional Begin Phoneme / End Phoneme pair representing the closing of the mouth.

Here is a simplified example of extracting speech synthesis information from audio markers. For a complete working example, refer to the MarkersCallback function in DemoSpeechEngine.cpp in the Wwise IntegrationDemo. 

const AkUInt32 kBeginWordMarkerID = 'bwrd';
const AkUInt32 kBeginPhonemeMarkerID = 'bphm';
const AkUInt32 kEndPhonemeMarkerID = 'ephm';
const AkUInt32 kEndWordMarkerID = 'ewrd';

static void MarkersCallback(AkCallbackType in_eType, AkEventCallbackInfo* in_pEventInfo, void* in_pCallbackInfo, void* in_pCookie)
{
	if (in_eType == AK_Marker)
	{
		AkMarkerCallbackInfo* pMarkerCallbackInfo = (AkMarkerCallbackInfo*)in_pCallbackInfo;

		AkUInt32 uMarkerID = pMarkerCallbackInfo->uIdentifier;
		AkUInt32 uMarkerPositionInAudioSamples = pMarkerCallbackInfo->uPosition;
		const char* szMarkerLabel = pMarkerCallbackInfo->strLabel;

		switch (uMarkerID)
		{
			case kBeginWordMarkerID:
			{
				std::string strMarkerLabel(szMarkerLabel);
				auto delimPos = strMarkerLabel.find(' ');
				if (delimPos != std::string::npos)
				{
					unsigned long uStartOfWord = std::stoul(strMarkerLabel.substr(0, delimPos));
					unsigned long uEndOfWord = std::stoul(strMarkerLabel.substr(delimPos));
				}
				break;
			}
			case kBeginPhonemeMarkerID:
			{
				AkInt16 visemeID = static_cast<AkInt16>(std::stoi(szMarkerLabel));
				break;
			}
			case kEndPhonemeMarkerID:
			case kEndWordMarkerID:
			{
				break; // No additional data in the label for these markers.
			}
		}
	}
}

Limitations

  • One voice per language at a time. Only one speechEngine voice per language or dialect can be played at one time. For example, you could play a voice for US English and another voice for British English, but not two voices for US English. If a second voice of the same language is triggered while one is already playing, the second voice is not initialized. An error message is logged to the Wwise Monitor and no audio is produced for that instance. Design your event system to avoid overlapping voices of the same language.

  • First-playback latency. The first time a specific voice is used after the application starts, the plug-in must load the voice model into memory, which causes an extended delay before audio begins. Subsequent playback requests using the same voice use the loaded model and respond with standard latency. If this initial delay is a problem for your project, you can trigger a silent warm-up event at startup to avoid it.

  • SSML syntax errors. If SSML is enabled and the input text contains invalid markup, the synthesis fails silently from the player's perspective: no audio is produced and an error message is logged to the Wwise Monitor. Validate SSML content before passing it to the plug-in.

  • RTPC changes mid-playback. Changes to Pitch, Speed, Sentence Pause, or Comma Pause via RTPC do not affect a synthesis that is already in progress. Updated values are applied the next time a new event is posted.
